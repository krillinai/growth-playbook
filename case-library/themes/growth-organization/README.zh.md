## 案例 1：Facebook / Meta - 围绕留存的跨职能责任

**环境。** 早期 Facebook 增长需要产品、工程、数据、营销与领导层连接社交价值、好友发现、激活、留存、埋点、实验与国际扩张。

**模式。** 公司级方向对齐团队，小型负责工作组处理有边界客户问题。增长团队拥有产品界面与数据权限，而不是作为只提供获客建议的职能存在。

```text
核心社交价值 -> 共享留存使用结果 -> 负责工作组
-> 产品与数据权限 -> 实验与决策 -> 可复用学习
```

**保留的证据。** 参与者讲述包括暂停路线图偿还 Data Debt、共同顶层指标、连接留存的激活代理、实验基础设施、Canonical Document 与清晰运营重点。可迁移经验是协调结果责任，而不是统一好友数阈值。

**迁移边界。** Facebook 的校园历史、网络效应、规模、数据、社会影响与市场时机都很特殊。公司整体结果不能归因到一个团队、领导者、指标或组织设计。

## 案例 2：Booking.com - 平台加团队的实验所有权

**环境。** 高产品与实验量带来一个选择：把决策集中到专家，还是通过共享质量标准给产品团队自助能力。

**模式。** 产品团队负责客户问题、假设与行动；中央实验能力提供基础设施、方法、指标、培训、文档和复杂案例支持。

```text
本地产品问题 -> 自助实验 -> 共享分组与指标
-> 本地解释与行动 -> 记录学习 -> 改善平台与实践
```

**保留的证据。** Lukas Vermeer 的讲述把分布式决策与共同护栏连接。并行实验通过曝光记录、合理交互分析、选择性互斥与升级管理，而不是全部进入中央队列。

**迁移边界。** 自助需要足够流量、埋点、统计判断、领导支持与可靠基础设施。小公司应针对已观察约束建设，而不是复制 Booking.com 规模系统。

## 案例 3：字节跳动 - 共享 UG 平台与本地执行

**环境。** 多产品、多市场公司重复面对获客、素材、归因、风控、召回与产品组合协调问题，同时产品和国家仍有实质差异。

**模式。** 共享 UG 能力处理稳定重复工作，产品与本地团队保留上下文执行。运营者定义目标、调查异常、适配市场，并把学习反馈到共享系统。

```text
产品与市场结果 -> 共享价值与投放服务
-> 产品或本地执行 -> Cohort 与风险证据
-> 更新分配、平台与市场决策
```

**保留的证据。** 公司页记录 UG 平台、预测 LTV、自动投放、素材系统、归因、风险隔离、产品组合互导、共享生命周期服务与本地适配。

**迁移边界。** 讲述跨越多年和多个产品。数据量、资金、推荐系统、监管环境、治理、本地团队与内部激励无法从平台模式中剥离。

## 案例 4：Ramp - 聚焦自治、增长工程与项目组合

**环境。** B2B 金融增长连接产品价值、企业控制、获客、账户扩张、经济性、销售、实验、品牌与内部自动化。

**模式。** 聚焦目标和上下文支持自治执行。增长工程连接产品、数据与 GTM 系统；书面推理暴露假设；实验组合与 Postmortem 指导分配并改善系统。

```text
客户与经济结果 -> 聚焦团队与决策权
-> 产品、增长工程与 GTM 执行 -> 有结论证据
-> 扩大、修改、停止或改善运营系统
```

**保留的证据。** Ramp 参与者讲述强调区分获客与扩张、把规模与回本和贡献毛利配对、产生有结论失败、跨周期分配投入、使用写作，以及让速度服务客户差异化。

**迁移边界。** 完整指标定义、实验日志、团队界面、经济性与 Agent 评估不可得。金融产品还需要限制自治的安全、审计、合规与风控。

## 案例 5：Pinterest - 组织随约束演进

**环境。** Pinterest 增长工作跨越 SEO 入口页、转化、新用户引导、留存、国际化、数据、实验与产品领导力。随着产品和问题成熟，所需所有权发生变化。

**模式。** Casey Winters 描述增长组织从通用中心团队向专业化所有权演进。团队需要拥有有意义结果和相关杠杆的执行权，同时不能永久累积所有产品界面。

```text
当前约束 -> 有边界负责人和所需杠杆 -> 证据与执行
-> 重复工作足够稳定后专业化 -> 新约束与所有权复核
```

**保留的证据。** Pinterest 证据连接真实入口界面所有权、定量实验、定性研究、下游质量、国际工作、数据能力、决策权与产品领导沟通。

**迁移边界。** 公开演讲没有提供完整团队时间线、人数模型、决策权地图或因果组织评估。搜索、产品、品牌、竞争与多个团队共同塑造结果。

## 跨公司模式

### 结构跟随约束

Facebook 围绕留存社交价值集中跨职能责任；Booking.com 在建立共享实验能力后分布产品决策；字节跳动集中重复 UG 服务并保留产品和市场上下文；Ramp 把聚焦自治与增长工程结合；Pinterest 随约束演进专业化所有权。

没有任何案例支持在定义结果和工作前先选择组织图。

### 共享能力应该赋能决策

Booking.com 与字节跳动展示两种平台加本地形式。共享层提供可靠共同服务，上下文团队负责客户与市场决策。Facebook 埋点与 Ramp 增长工程在不同范围遵循同一原则。

当共享团队成为权限队列、需求队列或脱离本地价值的全局优化器时，它会造成伤害。

### 自治需要上下文、证据与护栏

Facebook 小型工作组、Booking.com 自助团队、Ramp 聚焦团队与字节跳动本地执行，都依赖共享方向、可观察行为、明确边界与复盘。自治不等于没有标准。

### 组织需要保存学习与维护

埋点、实验记录、Canonical Document、Postmortem、平台反馈与所有权复核，把一次项目转化成可复用能力。每种模式也会产生需要负责人的维护与协调成本。

## 迁移矩阵

| 组织条件 | 最相关案例 | 可以借鉴 | 不能假定 |
| --- | --- | --- | --- |
| 一个主要客户价值约束需要多个职能 | Facebook / Meta | 共享结果、有边界工作组、产品和数据权限、清晰文档 | 一个指标或团队造成公司整体增长 |
| 多个产品团队需要可信常规实验 | Booking.com | 自助平台、共享方法、培训、本地决策责任 | 高实验量能够弥补薄弱战略或判断 |
| 产品与市场重复部分增长工作，但执行不同 | 字节跳动 | 集中稳定服务、委派上下文选择、嵌入风控与反馈 | 一个全局目标或自动化适合每个产品与市场 |
| B2B 增长跨产品、GTM、经济性与受监管运营 | Ramp | 聚焦目标、增长工程、项目组合、写作、有结论决策 | 速度或人才密度可以移除治理与维护需求 |
| 增长团队正在累积专业问题与界面 | Pinterest | 复核约束、专业化重复工作、保留有意义结果责任 | 成熟公司最终结构是正确起始结构 |

## 证据与局限

- 公司证据主要来自参与者访谈、公开演讲与重构公司页，而不是完整内部组织记录或因果评估。
- 案例跨越不同年代、规模、产品、市场、监管与增长工作定义。
- 人数、汇报线、决策权矩阵、依赖数据、绩效系统与组织变化结果都不完整。
- 公司整体结果还来自产品价值、市场条件、资金、战略、领导层、多个团队和未在此记录的机制。
- 收录用于描述运营模式，不代表认可每项公司决策、管理实践、劳动条件、产品影响或公开立场。
