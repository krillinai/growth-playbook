## 案例 1：Booking.com——作为共享产品基础设施的实验体系

Booking.com 把分流、曝光、分析、统一方法、指标、培训和文化结合，让产品团队无需把每项决策提交中央队列，就能运行标准实验。

```text
本地产品问题
-> 共享实验服务
-> 统一有效性与指标控制
-> 本地证据驱动决策
-> 文档化学习
```

可迁移的模式是“平台加本地所有权”。中央能力保护可靠性和可比性；产品团队继续拥有客户问题、假设与决策。当曝光可观察且重大交互得到适度管理时，并行实验可以提高吞吐量。

**主要边界：**小型组织可能没有足够实验量或流量支撑同样的平台复杂度。

## 案例 2：字节跳动——产品矩阵规模的增长平台

字节跳动案例覆盖预测 LTV、投放自动化、素材标签与生产、归因、风控、生命周期 SDK、跨产品流量和本地化。

```text
留存产品价值
-> 多周期价值预测
-> 市场与预算分配
-> 自动交付与素材学习
-> 归因与风险控制
-> 本地执行与模型更新
```

这是案例库中范围最广的基础设施案例，体现自动化、素材系统、价值模型、产品矩阵协同和治理为什么必须共同设计，也体现产品或市场语境显著不同的决策不适合强行中心化。

**主要边界：**系统和阈值主要来自前参与者回顾，不能视为官方架构或可直接迁移的 Benchmark。

## 案例 3：Facebook / Meta——先埋点，再优化

Facebook 早期增长案例把激活假设、留存用户分析、Cohort 指标、增长核算、新用户引导、实验与国际化连接起来。当关键流程无法被衡量时，埋点成为直接产品优先级。

```text
核心社交价值
-> 可观察激活行为
-> 埋点与 Cohort
-> 实验与引导改变
-> 留存与增长核算
-> 国际扩张
```

基础设施经验是数据采集应跟随决策模型。团队不需要记录所有事件，而是需要足够可信地观察激活、留存和边际用户变化。

**主要边界：**Facebook 拥有特殊规模、网络效应、数据量和产品权限。历史激活阈值不是通用目标。

## 案例 4：Ramp——连接产品、数据与 GTM

Ramp 证据覆盖增长工程、衡量、实验组合、复盘、产品速度、客户节省，以及通过共享上下文连接的新兴 AI Agent 运营。

这一模式体现相对紧凑的 B2B 增长栈：技术建设者连接产品与 GTM 数据，书面决策和复盘使学习可以复用。只有来源数据、权限、评估和人员所有权明确时，共享 AI 上下文才能降低协调成本。

**主要边界：**已有领导者与实践者叙述不构成完整系统图，也不是对 AI Agent 生产力的独立评估。

## 案例 5：Airbnb——产品化分发基础设施

Airbnb 历史上的 Craigslist 集成把重复分发工作流转化为产品能力，把房源字段、地区结构、发布步骤、回流路径和衡量连接到一条流程。

这个范围更窄的案例说明，增长基础设施不只包括中央数据平台。可复用产品界面也能减少重复渠道工作，并把分发连接到交易质量。

**主要边界：**案例不能证明集成的准确增量贡献，历史技术方法也不能在缺少当前授权、Consent 和平台政策审核时复制。

## 跨公司模式

### 中心化稳定重复工作

- Booking.com 中心化标准实验机制，同时保留本地产品决策。
- 字节跳动在多个产品间中心化重复投放、素材、归因与生命周期能力。
- Facebook 围绕激活和留存建设共享埋点与衡量。
- Ramp 通过工程和共享上下文连接重复的产品、数据与 GTM 工作。
- Airbnb 产品化一条重复外部分发流程。

共同原则不是最大程度中心化，而是建立稳定服务契约和清晰本地所有权。

### 基础设施跟随约束

| 约束 | 证据模式 |
| --- | --- |
| 实验请求在专家队列中等待 | Booking.com 的自助能力与共享护栏 |
| 手工投放与素材运营无法扩展 | 字节跳动的自动化与素材系统 |
| 团队无法可靠观察激活与留存 | Facebook 的埋点与增长核算 |
| 产品与 GTM 学习保持碎片化 | Ramp 的增长工程、写作与共享上下文 |
| 外部分发需要重复手工工作 | Airbnb 的产品化渠道流程 |

复制产物而不具备原始约束，会产生闲置平台与新依赖。

### 把活动连接到下游价值

实验数、交付量、生成资产数、消息数和平台用户数都是活动指标。只有这些活动连接到激活、留存、预订、客户节省、Margin 或更好决策时，案例才构成增长证据。

### 把风险放入架构

不同案例的风险形式不同：Booking.com 的统计有效性和交互；字节跳动的归因、作弊、权限与自动化；Facebook 的数据质量与组织激励；Ramp 的 AI 上下文、评估与所有权；Airbnb 的 Consent、可靠性与平台依赖。

风险控制应进入数据访问、分流、交付、模型与审核工作流，而不是只在末端出现。

## 迁移矩阵

| 组织条件 | 优先建设 | 避免 |
| --- | --- | --- |
| 早期产品、低流量、重复工作较少 | 关键事件、指标所有权、简单交付与决策日志 | 公司级平台或高级自动化 |
| 多个团队争议指标与实验 | 身份、语义定义、曝光完整性与质量诊断 | 没有共享计算的更多 Dashboard |
| 付费与素材吞吐量很高 | 成本采集、LTV 校准、素材血缘、疲劳与受限自动化 | 优化上游点击或产出数量 |
| 多产品或多市场 | 稳定共享服务、本地配置、产品矩阵增量与本地化 | 取消本地责任的单一全球目标 |
| B2B 产品与 GTM 碎片化 | 账户身份、产品到收入数据契约、工作流所有权与评估 | 没有客户价值模型就连接工具 |

## 证据与局限

- 证据主要来自参与者访谈、演讲和公开实践者分析，不是完整内部架构或因果评估。
- 公司实践跨越不同时间，不主张所有系统当前仍保持原样。
- 规模、品类、监管、数据量、流量、人才和市场结构限制直接迁移。
- 发布方增长、收入、客户、估值或实验量主张不用于证明基础设施有效性。
- 公司结果不能归因于某个平台、团队或实践者。
