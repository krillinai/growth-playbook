## 能力矩阵

| --- | --- | --- | --- | --- |
| [字节跳动](../../companies/bytedance/README.zh.md) | 现金、金币、优惠券、推荐奖励、Lite 产品 | 获客、重复活动、推荐与产品矩阵移动 | 奖励结束后的留存行为与产品矩阵增量价值 | 作弊、信号污染、蚕食与补贴依赖 |
| [Uber](../../companies/uber/README.zh.md) | 司机与乘客激励、本地保底、Promotion | 建立供给、需求、利用率与本地流动性 | 激励递减后的供需与交易质量 | 永久补贴、本地失衡、供给方经济性弱与逆向选择 |
| [Dropbox](../../companies/dropbox/README.zh.md) | 产品容量奖励，包括双边存储空间 | 合格推荐与更深产品使用 | 接收者激活、留存与持续存储价值 | 低质量邀请、滥用、服务成本与自然行为替代 |
| [Duolingo](../../companies/duolingo/README.zh.md) | Streak、里程碑、身份、提醒与进度恢复 | 重复学习练习与习惯连续性 | 有意义练习与不依赖强迫机制的回访 | 刷积分、通知疲劳、损失厌恶与学习质量偏移 |
| [Ramp](../../companies/ramp/README.zh.md) | 围绕客户节省的结构性商业模式对齐 | 客户更高效支出与运营时，公司增长仍保持可信 | 实际节省、信任、采用与持续经济性 | 缺少节省衡量的对齐主张或隐藏收入冲突 |

这些案例位于不同激励层。字节跳动和 Uber 使用可转移经济价值；Dropbox 授予产品价值；Duolingo 使用行为和保护机制；Ramp 是结构性对齐案例，不是奖励 Campaign。

## 案例 1：字节跳动——金钱奖励、Lite 产品与风险

字节跳动案例区分现金、虚拟金币、优惠券、推荐奖励和产品矩阵权益。这些机制具有不同接收者、成本、提现规则与行为影响。

```text
分群与目标行为
-> 奖励资格与交付
-> 获客、活动或推荐响应
-> 产品价值与留存行为
-> 作弊、蚕食与 Margin 检查
-> 继续、隔离、递减或停止
```

Lite 产品让产品矩阵影响更加明显。补贴版本可以触达不同分群或设备限制，也可能只把现有用户从更健康的产品移走。必须标记奖励行为，避免推荐和个性化系统把付款驱动活动误认为自然偏好。

**主要边界：**内部数字、阈值和系统描述主要来自前参与者回顾，不能作为可直接迁移的 Benchmark。

## 案例 2：Uber——用激励暂时协调流动性

Uber Marketplace 案例把激励视为本地供需形成的一项输入。司机可用性、乘客需求、利用率、接驾时间、履约、参与者经济性和留存共同作用于一个地域与时间窗口。

```text
本地流动性约束
-> 定向供给或需求激励
-> 可用性与交易提升
-> 参与者价值与利用率
-> 递减激励
-> 留存本地流动性与经济性
```

激励应针对受约束的一侧与转化，而不是最大化总注册。如果一个市场没有永久补贴就无法提供足够需求频率或供给方经济性，它可能存在结构性商业模式问题。

**主要边界：**Uber 的资本、劳动力池、城市运营、监管、规模与历史激励水平不是通用 Marketplace 启动模板。

## 案例 3：Dropbox——与产品价值一致的双边推荐

Dropbox 早期推荐案例奖励额外存储空间，把权益直接连接到产品核心价值。发送者和接收者都可以获得奖励，同时让接收者体验产品，而不是获得无关现金。

```text
已激活用户重视存储与同步
-> 邀请合格接收者
-> 接收者激活并使用产品
-> 授予产品容量
-> 双方继续获得存储价值
```

产品一致性不能消除经济性问题。存储有服务成本，推荐用户可能质量较低，部分接收者原本也会加入。需要评估接收者激活、留存、奖励成本、滥用和规模化边际表现。

**主要边界：**公开推荐叙述经过简化，没有提供完整实验设计、Cohort（同期群）经济性、基线和反事实结果。

## 案例 4：Duolingo——进度保护与有意义习惯

Duolingo 的 Streak 系统组合每日状态、可见进度、里程碑、提醒、庆祝、Streak Freeze 恢复和付费功能边界。激励同时来自身份和保护累计进度。

```text
有意义学习行动
-> 可见进度与 Streak 状态
-> 提醒、里程碑或恢复
-> 回到学习行动
-> 学习价值与持续习惯
```

可迁移的机制不是 Streak 数字本身，而是清晰进度、有限恢复、重复产品价值与实验之间的连接。护栏应覆盖学习质量、通知疲劳、强迫感和通过害怕损失来收费。

**主要边界：**Duolingo 是高频学习产品，拥有强识别品牌和大型实验体系，同一机制可能不适合低频或高风险产品。

## 案例 5：Ramp——结构性激励对齐

Ramp 的差异化主张是帮助客户节省金钱并提高财务运营效率，这与主要在客户增加支出或持有余额时受益的金融模式形成对比。

这不是推荐奖励或客户 Promotion，而是商业模式层的激励设计问题：

```text
客户节省金钱或运营时间
-> 产品创造可衡量价值
-> 采用与信任增加
-> 公司通过交付客户结果增长
```

只有定价、收入、产品行为和衡量都支持承诺时，对齐才可信。需要跟踪实际节省、客户结果、Margin、留存，以及可能重新制造冲突的收入机制。

**主要边界：**客户节省主张本身不能证明实际节省、增长因果贡献，也不能证明所有产品与收入来源完全对齐。

## 跨公司模式

### 让奖励匹配价值机制

- Dropbox 用存储容量奖励存储价值。
- Duolingo 围绕重复学习使用进度与恢复。
- Uber 在本地流动性受约束时付费协调供需。
- 字节跳动按照行为、分群和产品角色区分奖励类型。
- Ramp 围绕客户节省结果设计商业模式主张。

价值一致性提高机制合理性，但不能证明增量、持续性或正向经济性。

### 使用不同持续性检验

| 机制 | 持续性问题 |
| --- | --- |
| 现金或金币 | Payout 降低或停止后，行为是否保留？ |
| Marketplace 补贴 | 递减后本地流动性、质量与参与者经济性是否成立？ |
| 产品容量 | 推荐接收者是否激活、留存并持续重视产品？ |
| 进度保护 | 用户是否回到有意义行动，而不是只保护计数器？ |
| 结构性对齐 | 客户价值是否实际实现，并与公司增长建立经济连接？ |

### 衡量整个系统

激励可能改善目标指标，却伤害另一个产品、参与者侧、渠道、Cohort 或长期动机。应共同评估客户级、Marketplace 级和产品矩阵级结果。

### 扩大前设计风险

字节跳动强调反作弊与人群隔离；Uber 需要本地和跨边经济性；Dropbox 需要推荐验证与成本控制；Duolingo 需要伦理动机与学习质量护栏；Ramp 需要透明的结果与收入衡量。

## 迁移矩阵

| 情况 | 优先建设 | 避免 |
| --- | --- | --- |
| 产品价值尚未证明 | 客户研究、激活与留存证据 | 付费让用户忍受薄弱价值 |
| 推荐可能传递合格价值 | 接收者价值、发送者资格、激活、留存与滥用控制 | 只优化邀请数 |
| Marketplace 无法协调本地启动 | Hard Side 诊断、最小可行网络、受限补贴与递减测试 | 跨城市和时段使用统一激励 |
| 有价值习惯容易因偶发中断消失 | 清晰进度、有限恢复与价值驱动回访 | 强迫性损失、过度提醒或刷积分 |
| 多个产品共享用户 | 客户级 Holdout 与产品矩阵贡献 | 孤立评估被推广产品 |
| 商业模式主张客户对齐 | 实际客户结果、收入逻辑与冲突审核 | 把定位当成经济证明 |

## 证据与局限

- 证据主要来自参与者访谈、演讲和实践者分析，不是完整内部实验、Cohort、成本或作弊记录。
- 实践跨越公司不同时期，不代表当前所有项目。
- 公司规模、品类、监管、产品频率、数据、风控能力与参与者经济性限制迁移。
- 发布方增长、收入、客户、实验量或节省主张不用于因果证明。
- 公司结果不能归因于某项激励、团队或实践者。
