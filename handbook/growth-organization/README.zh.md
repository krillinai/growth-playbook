# 增长组织

> 增长组织围绕当前增长约束分配结果责任、决策权、能力与学习。正确结构会随产品、市场、阶段和工作变化，不存在通用增长组织图。

**[English](README.md)｜[简体中文](README.zh.md)**

## 定义与边界

增长组织是一套运营系统，决定谁负责增长结果、能够改变哪些杠杆、可以调用哪些能力、如何做出决策，以及学习如何越过单个项目被保留。

它不等于一个名为 Growth 的部门。

| 职能 | 核心责任 | 增长组织边界 |
| --- | --- | --- |
| 产品 | 客户问题、产品价值、体验、路线图 | 增长团队不应建立影子产品组织。 |
| 营销与分发 | 需求、定位、渠道、品牌、沟通 | 获客量不能补偿薄弱产品价值或留存。 |
| 销售与客户成功 | 资格判断、商业流程、实施、客户结果 | 产品信号不能替代权限、预算或关系判断。 |
| 数据与实验 | 衡量、因果学习、决策支持 | 中央专家赋能决策，不应拥有每项产品判断。 |
| 平台与基础设施 | 可复用系统、可靠性、治理、自助服务 | 共享团队不应成为无差别需求队列。 |
| 增长 | 围绕已诊断约束与可衡量客户结果协调责任 | 约束变化时，范围也应该变化。 |

组织可以没有永久增长团队，却仍然实践严谨增长；也可以拥有庞大增长部门，却没有清晰责任。

## 组织模型

```text
产品与市场环境
-> 已诊断增长约束
-> 负责的客户与业务结果
-> 所需杠杆与能力
-> 所有权与决策权
-> 执行、证据与复盘
-> 可复用学习与下一项约束
```

组织图是这个模型的输出，不是第一个输入。

## 设计输入

选择结构前，记录塑造结构的条件。

| 输入 | 核心问题 | 组织含义 |
| --- | --- | --- |
| 产品阶段 | 价值、PMF、可重复性还是规模是主要不确定性？ | 早期团队需要广泛学习，后期团队才可能适合专业化。 |
| 主要约束 | 获客、激活、留存、商业化、扩张还是系统容量？ | 所有权应该包含能够改变约束的杠杆。 |
| 产品结构 | 消费、B2B、Marketplace、网络、平台、产品组合还是 AI？ | 实体、依赖、风险与本地环境不同。 |
| 共享工作 | 哪些问题在团队或产品间重复出现？ | 稳定重复可以支持平台或卓越中心。 |
| 决策风险 | 哪些变化可逆、高风险、受监管或跨公司？ | 升级与复核应该随后果扩大。 |
| 证据成熟度 | 指标、Cohort、实验与定性学习是否可信？ | 没有证据的自治会规模化局部意见。 |
| 人才与容量 | 已具备哪些能力，瓶颈在哪里？ | 结构不能假定不存在的专家或管理带宽。 |
| 地域差异 | 哪些决策需要本地产品或市场知识？ | 共享标准可以与本地执行权共存。 |

## 结果责任

结果责任意味着一位负责人或团队能够解释目标客户状态、指标定义、当前证据、主要约束、取舍与下一项决策。它不意味着一个团队造成了整个公司结果。

```text
结果负责人
├── 客户状态与价值定义
├── 指标与护栏完整性
├── 优先约束
├── 所需杠杆访问权
├── 依赖与升级地图
└── 决策与学习记录
```

谨慎使用共享目标。公司级结果可以对齐团队，但每个贡献团队仍需要有边界的责任和可控输入。

## 决策权

决策权明确谁提出、决定、执行、复核，以及谁能够停止变化。它应该区分可逆、不可逆、本地、共享与受监管决策。

| 决策类型 | 默认负责人 | 必要控制 |
| --- | --- | --- |
| 可逆本地实验 | 结果团队 | 明确假设、指标、护栏与曝光记录 |
| 共享产品界面 | 指定跨团队负责人 | 界面契约、冲突处理、维护所有权 |
| 指标或平台标准 | 能力负责人加用户代表 | 版本、迁移、文档、例外流程 |
| 高风险客户或政策变化 | 负责高管或风险负责人 | 复核、分阶段曝光、回滚、事故方案 |
| 资源与组合分配 | 领导层与结果负责人 | 可比较证据、机会成本、明确取舍 |

共识不能替代决策权。可以广泛收集输入，同时保持最终决策负责人清晰。

## 运作维度

| 维度 | 核心问题 | 有用信号 |
| --- | --- | --- |
| 结果责任 | 谁对客户价值结果负责？ | 负责人清晰、定义稳定、决策及时 |
| 决策权 | 团队无需升级可以改变什么？ | 依赖等待、决策反复、升级负荷 |
| 能力访问 | 产品、数据、工程、设计、分发与运营能否共同工作？ | 从诊断到验证变化的时间 |
| 平台杠杆 | 哪些工作可复用并可自助？ | 采用、排队时间、重复建设、可靠性 |
| 学习系统 | 证据、失败与决策是否被复用？ | 决策落实与重复错误 |
| 组合质量 | 容量是否同时解决当前约束和未来选择？ | 结果贡献、学习、放弃工作 |
| 可持续性 | 速度是否保护质量、信任与人员？ | 事故、返工、流失、On-call 与会议负荷 |

## 所有权模式

| 模式 | 最适环境 | 优势 | 主要风险 |
| --- | --- | --- | --- |
| 中心增长团队 | 少数高价值约束需要集中专业能力 | 聚焦、共享上下文、快速形成能力 | 变成服务队列或脱离产品的优化团队 |
| 嵌入式增长责任 | 增长问题高度产品特定，团队具备必要技能 | 本地上下文强、责任持久 | 重复系统与标准不一致 |
| Growth-as-a-Service | 短期、有边界问题需要专家介入 | 灵活分配与快速诊断 | 交接薄弱，反复依赖专家 |
| 自主增长单元 | 重要结果需要专用界面、人员和快速迭代 | 责任清晰、决策快速 | 产品冲突、重复建设、维护债务 |
| 卓越中心 | 多团队需要方法、培训、治理与辅导 | 分布能力，不集中每项决策 | 只有建议影响，没有采用或结果责任 |
| 平台加本地混合 | 可复用基础设施服务不同产品或市场 | 同时获得规模收益与上下文执行 | 界面模糊与平台主导 |

应在定义结果、约束、杠杆、所需能力、决策权、周期与交接后选择。混合模式不自动更好，它最需要清晰界面。

## 中心、嵌入与混合边界

当输入、输出、可靠性要求与治理在团队间稳定时集中；当客户上下文、产品行为、市场条件或判断会实质改变决策时保持本地所有权。

| 通常共享 | 通常本地 |
| --- | --- |
| 身份、分组、事件标准、共同指标定义 | 客户问题与假设 |
| 可复用实验、消息、素材与交付服务 | 产品体验、市场信息与优先级 |
| 安全、隐私、风险与审计控制 | 本地市场、分群与渠道执行 |
| 文档、培训、模板与质量检查 | 客户上下文与取舍解释 |
| 产品组合可见性与容量标准 | 委派决策权内的最终行动 |

当本地团队绕过平台，或中央团队反复覆盖上下文时，应重新检查边界。

## 增长团队与职能边界

先定义客户与结果，再分配职能贡献，避免建立重复路线图。

```text
共享结果
├── 产品：价值路径与体验
├── 工程：产品界面与可靠系统
├── 数据：定义、诊断、因果学习
├── 设计与研究：行为、理解、信任
├── 营销：需求、定位、分发、证明
├── 销售与成功：资格、实施、账户价值
└── 运营与风险：容量、政策、质量、事故
```

跨职能不意味着所有职能参加每场会议。工作团队应包含当前决策所需能力，并明确其他职能的咨询与升级方式。

## 能力地图

| 能力 | 核心责任 | 集中化问题 |
| --- | --- | --- |
| 增长产品管理 | 约束选择、结果责任、取舍 | 工作是产品特定还是跨界面共享？ |
| 增长工程 | 产品变化、内部工具、集成、自动化 | 重复工作是否足以支持平台或专用团队？ |
| 数据科学与分析 | 指标完整性、诊断、因果与预测证据 | 哪些标准必须共享，哪些分析需要上下文？ |
| 设计与研究 | 客户理解、路径设计、定性证据 | 专家能否足够接近客户决策？ |
| 分发与生命周期 | 渠道、内容、消息、CRM、销售或合作动作 | 机制是否需要产品权限或本地市场能力？ |
| 实验 | 设计、分组、分析、解释、文档 | 标准实验能否自助，复杂工作是否得到复核？ |
| 运营与赋能 | 流程、容量、培训、文档、供应商与市场执行 | 哪些协调工作真正降低依赖成本？ |
| 治理与风险 | 隐私、安全、法律、信任、财务与运营控制 | 哪些决策必须复核或拥有停止权？ |

头衔不如这些能力能否作用于约束重要。

## 增长工程

增长工程应该有明确客户和结果。它可以建设产品界面、实验服务、数据流、销售工具、素材系统、生命周期交付与自动化。

当请求缺少共享约束与决策框架时，它会变成需求队列；当它拥有广泛基础设施，却不负责长期可靠性、采用或维护时，它会变成影子平台团队。

应衡量从诊断到验证影响的时间、复用、可靠性、阻塞依赖、支持负担与被移除工作，而不只是发布数量。

## 卓越中心与自助服务

卓越中心应该通过标准、工具、辅导、审计、案例与升级路径，让可信路径更容易，而不应要求专家手工批准每项常规决策。

```text
共同方法与平台
-> 有文档的自助路径
-> 本地团队负责问题与行动
-> 专家支持例外与复杂风险
-> 学习改善标准与工具
```

衡量采用、决策质量、例外数量、排队时间、培训效果、重复错误，以及证据是否改变决策。

## 平台与本地所有权

平台加本地模式适合产品组合和国际增长。共享团队可以提供身份、归因、实验、交付、素材、生命周期、风险与报告服务；产品或市场团队保留客户、渠道、内容与运营上下文。

定义服务契约、本地覆盖、决策日志、升级、成本分配、可靠性目标，以及本地反馈如何进入平台路线图。当一个全局目标抹去产品或市场差异时，集中化会造成伤害。

## 人员配置与团队组成

从工作开始，而不是从标准人数比例开始。

| 需要 | 人员含义 |
| --- | --- |
| 客户或价值不清 | 优先配置强产品、研究与定性能力，再考虑优化专家 |
| 埋点与因果缺口 | 高实验量前先增加数据与工程能力 |
| 产品路径约束 | 配置拥有界面的产品、设计、工程与分析 |
| 渠道或市场约束 | 配置分发、素材、本地运营、产品与经济性能力 |
| B2B 账户扩张 | 配置产品、销售、成功、数据、实施与治理 |
| Marketplace 流动性 | 配置产品、运营、本地市场、激励、风险与双边分析 |
| 共享系统约束 | 配置平台工程、产品管理、可靠性、赋能与治理 |

小团队需要能力面广且拥有一到两项深度的人。大团队可以专业化，但交接和管理层级会增加协调成本。

## 招聘与评估

针对缺失能力和决策环境招聘，而不是招聘通用“增长黑客”。评估：

- 客户与业务理解；
- 结构化诊断与因果推理；
- 至少一项相关能力的深度；
- 跨职能工作但不隐藏责任的能力；
- 对指标、经济性、信任与取舍的判断；
- 书面与口头清晰度；
- 从失败或模糊工作中学习；
- 对运营、法律与伦理约束的尊重。

Work Sample 应接近真实问题：诊断约束、定义证据、做出决策，并说明什么信息会改变建议。

## 领导与管理

领导者提供方向、上下文、约束、决策权，并保护团队不受碎片化需求干扰。他们应该让分歧有用，并避免高层直觉在没有解释时覆盖证据。

| 领导责任 | 运营行为 |
| --- | --- |
| 方向 | 定义客户价值、战略与少量结果。 |
| 上下文 | 共享证据、经济性、风险、依赖与取舍。 |
| 自治 | 在明确边界内委派可逆决策。 |
| 质量 | 要求清晰定义、护栏与因果谦逊。 |
| 冲突 | 指定决策负责人并记录处理结果。 |
| 学习 | 奖励结论、停止工作与改变决策，而不只奖励正向结果。 |
| 可持续性 | 管理工作量、事故、维护与关键人员风险。 |

没有上下文的自治会造成偏离；没有决策权的上下文只会产生等待。

## 项目优先级

根据当前约束、绝对预期贡献、证据、成本、时间、依赖、可逆性与战略选择价值排序。

```text
预期贡献 = 合格 Reach x 可能 Impact x Confidence
```

公式是讨论工具，不是虚假精确。在极小界面上的高相对提升，可能不如影响多数合格用户的小幅改善。

项目组合包括：

- 保护和改善已验证价值的核心优化；
- 扩展当前模式的相邻投入；
- 测试不同机制的探索投入；
- 提高未来决策或执行能力的系统投入。

每项工作都需要负责人、决策、证据周期、护栏、依赖与停止条件。

## 实验与项目组合

实验数量不是目标。健康组合解决重要不确定性并改变资源分配。

| 组合问题 | 有用证据 |
| --- | --- |
| 项目是否连接主要约束？ | 按结果与约束划分的容量 |
| 工作是否产生结论？ | Win、Loss、无实质影响或未解决；决策是否变化 |
| 风险是否平衡？ | 优化、相邻、探索、平台与维护组合 |
| 依赖是否可见？ | 等待时间、阻塞天数、升级、共享界面冲突 |
| 学习是否复利？ | 被复用的发现、更新标准、避免重复实验 |
| 旧工作是否移除？ | 被移除的实验、工具、Flag、报告与流程 |

## 运营节奏

使用能够做出决策并保存学习的最少定期机制。

| 节奏 | 目的 | 输出 |
| --- | --- | --- |
| 周度或工作复盘 | 检查约束、证据、阻塞与决策 | 明确决策与负责人 |
| 实验或项目读数 | 解释结果与下一步行动 | 发布、停止、修改、扩展或调查 |
| 组合复盘 | 在不同结果和周期间重新分配容量 | 更新优先级与停止工作 |
| 指标复盘 | 对账结果、输入、质量与可持续性 | 定义或行动变化 |
| Postmortem | 解释假设、事件、影响与系统变化 | 预防、恢复或流程改善 |
| 周期战略复盘 | 重新检查产品、市场、模式与主要约束 | 新组织或组合假设 |

会议应该产生决策，而不是复述状态。

## 写作与决策记录

书面产物让假设、替代方案、证据、所有权与取舍可以被检查。一个决策使用一份 Canonical Record，而不是几份相互竞争的文档。

```text
问题与客户状态：
需要的决策：
当前证据与不确定性：
选项与取舍：
建议：
负责人和决策权：
指标、护栏与周期：
依赖与回滚：
决策与后续：
```

文档应该降低协调成本。当写作替代客户接触或决策时，它会造成伤害。

## 组织健康指标

| 层级 | 指标 | 解释 |
| --- | --- | --- |
| 结果 | 按负责范围观察客户与业务结果变化 | 归因到一个团队仍然有限。 |
| 决策 | 决策时间、决策反复、证据采用 | 快速错误决策不是健康。 |
| 执行 | 从诊断到曝光与验证结果的时间 | 把主动工作与依赖等待分开。 |
| 学习 | 有结论工作、复用学习、重复错误 | 负向结果在改变行动时仍有价值。 |
| 平台 | 采用、自助率、可靠性、队列时间、重复建设 | 高使用量可能来自锁定而非价值。 |
| 组合 | 按周期划分的容量、停止工作、维护负担 | 更多项目可能降低吞吐。 |
| 可持续性 | 事故、返工、On-call、会议负荷、人员流失 | 谨慎解释并保护隐私。 |

不要建立隐藏底层取舍的组织综合分数。

## 激励与绩效管理

奖励长期客户与业务结果、决策质量、协作、学习与系统改善。避免只按照实验 Win、功能发布、线索量或局部 KPI 变化激励。

绩效评估应该区分可控贡献与公司整体结果，否则团队可能操纵指标、回避不确定工作、隐藏失败，或把质量和维护成本转嫁给他人。

## 可持续性与维护

增长工作会产生代码、Flag、活动、数据管道、模型、内容、供应商、权限与运营承诺。为它们分配长期所有权或移除日期。

衡量维护、事故、支持、合规与认知负担。累积无主系统的速度会降低未来决策容量。

## 问题诊断

| 可观察模式 | 下一步检查 | 可能的应对 |
| --- | --- | --- |
| 想法多，交付慢 | 决策权、依赖、界面所有权 | 给一个团队有边界所有权或移除审批。 |
| 实验快，结果变化小 | 约束选择、Reach、指标层级 | 从局部优化转向主要约束。 |
| 中央团队变成队列 | 服务边界、自助、重复需求 | 产品化稳定工作，把判断留给本地团队。 |
| 嵌入团队重复建设 | 重复基础设施、标准、可靠性 | 建立拥有清晰客户的平台或卓越中心。 |
| 平台采用率低 | 用户需要、工作流匹配、迁移与支持 | 缩窄范围，与产品团队重建。 |
| 增长与核心产品冲突 | 共享结果、界面权限、维护 | 明确所有权并使用客户价值护栏。 |
| 团队等待共识 | 决策负责人、可逆性、升级 | 把咨询与最终权力分开。 |
| 高速度伴随事故或流失上升 | 工作量、质量债务、人员、激励 | 降低负荷并恢复维护与安全容量。 |
| 全球方法在本地失败 | 市场上下文、本地权力、平台目标 | 委派执行，把本地学习反馈到共享系统。 |

这些模式用于缩小调查范围，不能直接证明根因。

## 操作方法

1. 定义客户与业务结果、指标、护栏和当前约束。
2. 描绘所需杠杆、能力、依赖、共享系统与风险。
3. 选择所有权模式与预计周期。
4. 指定一位结果负责人并明确决策权。
5. 给团队相关产品、数据、分发与运营杠杆的访问权。
6. 建立组合、决策、指标与 Postmortem 节奏。
7. 衡量结果贡献、决策速度、依赖等待、学习、质量与可持续性。
8. 只有稳定共同需求出现后才产品化重复工作。
9. 当约束、阶段或共享能力变化时交接或重组。

## 重组触发条件

只有责任与决策流不再匹配工作时，重组才合理，而不是增长放缓就重组。

| 触发条件 | 可能结构变化 |
| --- | --- |
| 一个约束占主导且没有负责人 | 建立有边界跨职能任务团队。 |
| 多团队重复同一项可靠工作 | 建立共享平台或赋能能力。 |
| 中央队列拖慢上下文决策 | 嵌入所有权并提供自助标准。 |
| 嵌入重复建设制造可靠性风险 | 集中稳定基础设施与治理。 |
| 产品或市场组合多元化 | 使用平台加产品或平台加本地界面。 |
| PMF 或战略变化 | 解散、转向或重建增长职责。 |
| 维护与信任债务上升 | 把容量转向可靠性、治理与移除。 |

每次重组都有迁移成本。定义哪些决策会改善、哪些工作停止、谁负责未完成工作，以及何时复核新模式。

## 成熟度模型

| 阶段 | 特征 | 下一项约束 |
| --- | --- | --- |
| 1. 英雄项目 | 增长依赖个人、活动与非正式权限 | 定义结果、所有权、指标与决策权。 |
| 2. 跨职能团队 | 有边界团队可以作用于一个主要约束 | 建立可重复节奏与能力访问。 |
| 3. 多团队系统 | 嵌入或专业团队共享标准与组合 | 减少重复，并产品化稳定能力。 |
| 4. 平台加本地 | 共享服务通过明确界面赋能上下文团队 | 改善治理、分配、采用与本地反馈。 |
| 5. 自适应组织 | 结构、容量与权力根据约束和证据变化 | 保持清晰、韧性、信任与人工判断。 |

成熟度不要求经历每种结构。只有一个团队的小产品可能一直不需要增长平台。

## 建设顺序

1. 命名团队前先诊断约束。
2. 定义客户结果、业务边界、指标与护栏。
3. 描绘杠杆、能力、依赖、风险与本地上下文。
4. 选择能够负责结果的最小结构。
5. 指定一位负责人并明确可逆与不可逆决策权。
6. 根据需要提供产品、数据、工程、设计、分发与运营访问权。
7. 建立项目组合、书面决策、结果读数与 Postmortem。
8. 同时衡量决策与依赖流，以及结果变化。
9. 把稳定重复工作转化成拥有清晰客户的自助能力。
10. 当约束、阶段、产品、市场或维护负担变化时复核职责。

## 常见错误

- 招增长团队来制造产品市场匹配。
- 缺少相同规模、流量、平台或人才时复制大公司架构。
- 命名共享目标，却没有有边界责任和可控输入。
- 建立中心需求队列，而不是可复用自助能力。
- 嵌入团队却没有共享定义、平台或治理。
- 奖励实验 Win、产量或局部指标，而不是决策与长期价值。
- 把共识当成决策所有权。
- 在诊断约束与决策流问题前重组。
- 忽略维护、信任、风险与团队可持续性。
