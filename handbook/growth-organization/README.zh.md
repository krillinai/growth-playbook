# 增长组织

> 增长组织围绕当前增长约束分配责任、决策权、能力与学习，不存在通用的增长组织架构。

**[English](README.md)｜[简体中文](README.zh.md)**

## 组织模型

```text
增长约束 -> 负责的结果 -> 跨职能团队
         -> 决策权 -> 共享能力
         -> 复盘与学习 -> 下一个约束
```

常见形态包括中心团队、嵌入团队、职能卓越中心，以及平台加本地执行的混合模式。选择取决于产品阶段、主要约束、依赖关系与共享能力。

## 运作维度

| 维度 | 核心问题 | 信号 |
| --- | --- | --- |
| 结果责任 | 谁对客户价值结果负责？ | 明确负责人和决策 |
| 决策权 | 团队无需升级可以改变什么？ | 依赖等待与决策反复 |
| 能力访问 | 产品、数据、工程、设计和分发能否共同工作？ | 周期与阻塞工作 |
| 学习系统 | 证据与失败是否被复用？ | 后续落实与重复错误 |
| 可持续性 | 速度是否保护质量与人员？ | 事故、返工、流失 |

## 问题诊断

| 模式 | 下一步检查 |
| --- | --- |
| 想法多，交付慢 | 依赖、产品权限、决策权 |
| 实验快，业务变化小 | 约束选择薄弱或责任碎片化 |
| 中心团队变成排队入口 | 过度中心化和自助边界不清 |
| 嵌入团队重复建设 | 缺少共享平台或标准 |
| 增长损害产品质量 | 结果错位和护栏缺失 |

## 操作方法

1. 明确当前约束与需要负责的客户价值结果。
2. 绘制所需职能、依赖和共享能力。
3. 指定一位负责人，明确可逆和不可逆决策权。
4. 设置结果、输入、质量与可持续性护栏。
5. 用书面复盘改进系统，而不是归咎个人。
6. 约束或产品阶段变化时重新审视组织。

## 常见错误

- 招增长团队来制造产品市场匹配。
- 缺少相同规模与平台时复制大公司架构。
- 没有角色边界就让每位员工都做营销。
- 奖励实验数量而不是客户与业务结果。
