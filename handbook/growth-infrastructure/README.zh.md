# 增长基础设施

> 增长基础设施把重复的数据、交付、生命周期、素材与决策工作变成受治理的可复用系统。

**[English](README.md)｜[简体中文](README.zh.md)**

## 能力模型

```text
共同问题 -> 稳定接口 -> 可复用服务 -> 本地决策
         -> 日志与控制 -> 反馈 -> 平台改进
```

稳定能力适合中心化；产品和市场上下文会改变决策时，需要保留本地判断。

## 层级与指标

| 层级 | 示例 | 衡量 |
| --- | --- | --- |
| 数据 | 事件、身份、Cohort、归因 | 新鲜度、完整性、事故 |
| 决策 | LTV、分配、实验服务 | 准确性、校准、人工覆盖率 |
| 交付 | 消息、投放、本地化 | 周期、成功率、采用率 |
| 素材 | 资产、生成、审核、复用 | 吞吐、质量、疲劳 |
| 治理 | 权限、日志、风控、停止控制 | 违规、可审计性、恢复时间 |

## 中心化判断

多个团队重复同一问题、输入输出稳定、共享质量优于本地方案且治理明确时适合中心化。上下文主导、流程快速变化或协调成本高于复用收益时应保留本地能力。

## 操作方法

1. 盘点重复工作与内部使用者。
2. 定义服务契约、负责人、权限、日志、覆盖与停止条件。
3. 用一个真实工作流试点，并衡量下游客户价值。
4. 跟踪采用、周期、可靠性、决策质量和本地绕行。
5. 下线无人使用的能力，并保留退出路径。

## 常见错误

- 自动化不稳定流程；中心化所有决策；只看内部使用不看客户结果；为不存在的规模或监管提前建设。
