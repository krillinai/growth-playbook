# 增长基础设施

> 增长基础设施把重复的衡量、决策、交付、素材和治理工作转化为可复用系统，同时保留产品与市场所需的本地判断。

**[English](README.md)｜[简体中文](README.zh.md)**

## 定义与边界

增长基础设施是支持团队在各增长环节观察客户行为、统一指标、运行决策、交付干预、学习结果并控制风险的共享技术与运营能力。

它不是一组 Dashboard、一个大型数据仓库，也不是审批所有 Campaign 的中央团队。只有重复问题具备稳定输入、输出、质量要求和多个内部使用者时，基础设施才有价值。

```text
客户与产品事件
-> 可信身份和数据
-> 统一指标与决策服务
-> 实验、模型与资源分配
-> 交付与素材执行
-> 结果、控制与反馈
-> 可复用的组织学习
```

当上下文无法被安全标准化时，产品战略、市场语境、客户研究、创意判断和结果责任仍应留在本地团队。

## 能力架构

| 层级 | 核心责任 | 典型能力 | 主要故障 |
| --- | --- | --- | --- |
| 数据基础 | 产生可信行为记录 | 事件分类、身份、Consent、Cohort（同期群）、质量监控 | 团队优化不完整或不一致的数据 |
| 指标系统 | 让结果可以比较 | 语义定义、North Star、输入指标、护栏、指标注册表 | 同一个指标名在不同团队含义不同 |
| 实验服务 | 安全估计因果效应 | 分流、曝光、Feature Flag、SRM（样本比例失配）检查、Scorecard、Holdout | 平台产生精确但无效的结果 |
| 决策服务 | 支持重复资源分配 | 分群、归因、LTV（客户生命周期价值）、预测、排序、预算规则 | 未经校准就把模型输出当成事实 |
| 执行系统 | 可靠交付干预 | 付费投放、消息、生命周期编排、本地化 | 自动化放大错误定向或客户伤害 |
| 素材系统 | 把资产转化为可复用学习 | 资产注册、标签、生成、审核、疲劳、效果反馈 | 变体数量替代有意义的假设 |
| 治理与运营 | 让系统保持可控 | 权限、日志、审批、人工覆盖、停止控制、事故响应 | 规模带来的风险快于学习增长 |

各层通过契约连接，但必须能够独立诊断。复杂实验界面无法弥补身份错误，准确的 LTV 模型也无法修复不可靠的交付 Pipeline。

## 1. 数据基础

### 事件模型

围绕有意义的客户和系统状态定义事件，而不是记录所有可用点击。

```text
事件名称
实体与行为主体
时间戳与来源
属性与版本
资格与曝光语境
Consent 与保留等级
负责人和质量要求
```

区分意图、动作、结果与质量。例如 `checkout_started`、`payment_submitted`、`payment_succeeded` 和 `order_refunded` 回答的是不同问题。

### 身份与资格

增长决策可能横跨匿名访客、注册用户、账户、家庭、设备、企业、卖家和市场。明确身份如何合并、拆分、过期和跨设备连接。实验分流身份、分析身份、计费身份和消息触达资格的规则不同时，应分别处理。

身份错误会导致用户重复、选择性数据缺失、错误归因、过度触达、SRM 和虚假留存。

### 数据质量

监控完整性、新鲜度、有效性、唯一性、一致性与血缘。每个影响决策的关键事件都应有负责人、预期数量或比例、Schema 版本、下游使用者和事故处理路径。

根据失败成本使用不变量指标、与运营系统对账、自动异常检测和可重放的数据转换。

## 2. 指标系统

指标系统把原始事件转化为稳定的决策语言。

| 组成 | 作用 |
| --- | --- |
| 指标定义 | 公式、实体、分母、时间窗口、排除项、货币与负责人 |
| 语义层 | 供 Dashboard、实验、模型和复盘共同使用的可复用计算 |
| 指标树 | 把客户价值和业务结果连接到可控制输入 |
| 护栏 | 保护质量、信任、可靠性、Margin、生态和合规 |
| 版本 | 数据来源或逻辑变化后仍能保留含义 |
| 注册表 | 让定义、负责人、依赖和批准用途可以被发现 |

不能因为指标名称相同就强行中心化。留存消费者、活跃 Workspace、完成行程和留存商家分别需要符合产品价值的定义。

指标设计与复盘参见[指标与衡量](../metrics/README.zh.md)。

## 3. 实验服务

实验服务应覆盖从资格判断到决策的完整路径：

```text
资格
-> 确定性分流
-> Treatment 交付
-> 曝光记录
-> 身份与排除
-> 指标计算
-> 质量诊断
-> 效应解释
-> 决策与学习记录
```

核心服务可以包括 Feature Flag、互斥组、并行曝光日志、样本规划、SRM 检测、不变量检查、Scorecard、长期 Holdout 和实验知识库。

自助式实验不等于取消统计标准。标准实验应该易于运行；特殊随机化单位、干扰、高风险改变、自适应分配或结果成熟很慢的实验需要更强审核。

因果设计与决策规则参见[实验系统](../experimentation/README.zh.md)。

## 4. 决策服务

决策服务适合部分标准化那些足够频繁的选择。

### 分群

区分描述性分群和决策分群。有效的决策分群会改变资格、体验、渠道、定价、服务等级或资源分配，并且足够稳定，可以被运营。

### 归因

归因分配可观察的 Credit；增量衡量干预实际造成了什么。应区分运营报表、结算对账、多触点叙事和因果衡量。任何归因模型都不能消除对实验、Holdout、地域测试或明确假设的需要。

### LTV 与预测

明确预测周期、收入或贡献口径、留存假设、获客和服务成本、不确定性以及校准方法。用成熟 Cohort 对照预测，并保留现金、Margin 与 Stop-Loss 限制。

### 资源分配与自动化

模型可以推荐受众、预算、Bid、素材、时机或下一步动作。在自动化前定义允许输入、目标、护栏、置信度、人工覆盖权、日志、Fallback 和重新训练条件。

## 5. 执行系统

执行系统通过付费渠道、产品界面、Email、Push、消息、销售和本地市场，把决策转化为客户可见的动作。

可复用能力包括资格、抑制、频控、排期、本地化、Channel Adapter、重试、幂等、交付回执、成本采集和结果反馈。

交付成功不等于客户成功。除了发送、曝光、点击与花费，还要衡量下游激活、留存价值、Margin、投诉、退订和渠道饱和。

## 6. 素材系统

素材基础设施应保留客户证据、Concept、资产、受众、Placement、曝光和下游结果之间的关系。

| 能力 | 支持的决策 |
| --- | --- |
| 资产注册与血缘 | 创建、修改、批准和复用了什么？ |
| 结构化标签 | 使用了什么 Persona、Pillar、Angle、Proof、Format、Hook、产品、Creator 与受众？ |
| Concept 血缘 | 哪些资产属于同一假设的迭代，哪些属于 Net-new Concept？ |
| 生成与适配 | 哪些变体能在品牌和政策规则内生成？ |
| 审核流程 | 谁检查主张、权利、安全、本地化与质量？ |
| 疲劳监控 | 重复曝光何时降低效果或信任？ |
| 结果反馈 | 哪种创意机制贡献了留存客户价值？ |

```text
研究与账户证据
-> Concept 地图与优先 Brief
-> 生产、Creator 权利与审核
-> 上线、投放与曝光
-> Concept 级下游分析
-> 决策日志与下一份 Brief
```

指定一名负责人维护 Concept 地图，并让证据在研究、品牌、Creator、编辑、媒体采购、落地路径与分析之间流动。素材系统不能以资产数量为目标。有效吞吐量是通过可信下游衡量测试了多少会影响决策的假设。

## 7. 治理与运营

治理应内置在系统中，而不是成为最后一道审批。

| 控制 | 必须回答的问题 |
| --- | --- |
| 权限 | 谁能读取数据、改变逻辑、启动动作和批准例外？ |
| 审计 | 能否还原输入、版本、决策与输出？ |
| 人工覆盖 | 谁能干预、需要什么证据、持续多久？ |
| 停止控制 | 能否快速停止有害投放、花费、Treatment 或模型行为？ |
| 隐私与 Consent | 每种用途是否符合客户授权与保留政策？ |
| 风险隔离 | 激励机制是否允许建设者、运营者和审核者相互挑战？ |
| 事故响应 | 是否定义等级、负责人、沟通、恢复与复盘？ |

本地团队需要为特殊语境保留 Escape Hatch，但所有覆盖都必须可见、可审核。

## 中心化判断

以下多数条件成立时，能力适合中心化：

- 多个团队反复解决同一个问题；
- 输入、输出和服务要求稳定；
- 共享质量或合规优于本地方案；
- 平台协调成本低于重复建设；
- 本地语境可以通过配置或明确接口进入；
- 有明确负责人维护可靠性和持续演进。

当流程快速变化、产品语境决定答案、使用量不足以支撑服务，或中央所有权会形成决策队列时，应保留本地能力。

## 自建、采购或组合

| 选择 | 适用情况 | 主要风险 |
| --- | --- | --- |
| 采购 | 标准能力、差异化较低、需要快速上线、供应商成熟 | 锁定、数据边界、价格与定制限制 |
| 自建 | 战略差异化、特殊规模或流程、内部所有权强 | 长期维护、可靠性、人员与隐性机会成本 |
| 组合 | 标准核心加差异化内部逻辑 | 集成复杂度与事故责任不清 |

评估功能适配、数据访问、延迟、可靠性、隐私、扩展性、总成本、迁移路径、可观察性与退出成本。自建需要长期运营负责人，而不只是实现团队。

## 成熟度模型

| 阶段 | 特征 | 下一项约束 |
| --- | --- | --- |
| 1. 碎片化 | 手工导出、事件不一致、本地 Dashboard、Campaign 专用逻辑 | 建立关键事件、负责人和对账 |
| 2. 标准化 | 统一事件分类、指标定义、基础交付与实验服务 | 提高可靠性、可发现性与采用 |
| 3. 自助化 | 团队无需排队即可访问受治理数据并运行标准决策 | 增加诊断、培训、血缘与更强护栏 |
| 4. 自动化 | 重复分配与交付决策进入反馈循环 | 控制模型风险、漂移、人工覆盖和客户伤害 |
| 5. 自适应平台 | 多产品与市场复用能力并保留本地控制 | 防止复杂度、锁定、产品矩阵冲突和责任丢失 |

成熟度不是目标。聚焦型公司可能长期适合阶段 2 或 3。只有当前约束和复用价值能够证明新增系统成本时才继续升级。

## 健康度指标

### 可靠性

- 数据新鲜度、完整性、对账误差和事故率；
- 分流与曝光完整性；
- 交付成功率、延迟、重试率与恢复时间；
- 模型校准、漂移和 Fallback 率。

### 采用与效率

- 活跃内部使用者与重复使用；
- 从问题到可信决策的时间；
- 自助完成比例与支持队列比例；
- 重复 Pipeline 与本地绕行；
- 每项有效决策或留存结果的平台成本。

### 决策质量

- 预先定义指标和护栏的决策比例；
- 真正改变决策的证据；
- 后续团队复用实验与模型发现的比例；
- 预测与实际结果的校准；
- 人工覆盖质量和决策后复盘。

### 客户与风险结果

- 激活、留存、Margin 与客户价值影响；
- 投诉、退订、作弊、政策违规与蚕食；
- 隐私、权限与审计事故；
- 上线前后发现的伤害。

内部采用是必要条件，但并不充分。被广泛使用却无法改善决策质量或客户价值的平台，只产生基础设施活动，不产生增长杠杆。

## 诊断地图

| 症状 | 优先检查的问题 | 第一份证据 |
| --- | --- | --- |
| 不同团队报告不同数字 | 事件、身份、分母和窗口定义是否统一？ | 指标血缘与 Query 对比 |
| 实验经常被质疑 | 分流、曝光、SRM、指标计算和决策规则是否可信？ | A/A Test、不变量与 Scorecard 审核 |
| Campaign 数量上升但留存价值不变 | 交付和素材反馈是否连接激活、留存与 Margin？ | Cohort 与增量分析 |
| 中央团队成为队列 | 哪些标准决策可以通过护栏变成自助服务？ | 请求清单与周期拆解 |
| 自动化结果剧烈波动 | 目标、训练数据、漂移、人工覆盖与停止控制是否明确？ | 决策日志与 Back-testing |
| 本地团队绕过平台 | 接口是否缺少必要语境、太慢或不如本地方案可靠？ | 绕行访谈与服务遥测 |
| 全球上线效果不足 | 哪些数据、渠道、素材、监管或市场假设没有本地化？ | 市场级漏斗与运营对比 |
| 平台成本快于采用增长 | 哪些能力缺少重复使用者或可衡量决策价值？ | 成本采用比与下线复盘 |

## 建设顺序

1. 明确受约束的增长决策和客户结果。
2. 绘制当前工作流、系统、手工工作、负责人和故障模式。
3. 修复关键事件、身份、指标与权限基础。
4. 选择一个高价值、重复出现且有真实内部使用者的工作流。
5. 定义服务契约、质量目标、日志、人工覆盖与停止条件。
6. 端到端试点，并衡量决策时间、可靠性、采用和下游价值。
7. 记录本地团队与平台团队的所有权边界。
8. 只有第一个工作流获得信任后，才增加自助、自动化和更多使用者。
9. 定期审核闲置能力、绕行、事故与平台经济性。

## 常见错误

- 在定义决策和所有权前购买工具。
- 记录所有事件，却没有关键数据契约。
- 把一个 Dashboard 当成语义层。
- 在复用计算的同时把产品判断也中心化。
- 自动化不稳定、有偏或不可审计的流程。
- 优化发送、花费、实验或资产，而不是留存价值。
- 在需求出现前按照另一家公司的规模建设。
- 只衡量平台采用，不衡量可靠性、决策、客户与风险结果。
- 因为迁移不方便而保留无法创造价值的遗留服务。
