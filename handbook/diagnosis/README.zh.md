# 增长诊断协议

> 把增长症状转化为有证据支持的核心约束、聚焦行动方案与正确执行路径的可重复方法。

**[English](README.md)｜[简体中文](README.zh.md)**

## 协议解决什么问题

增长团队经常从可见症状开始：流量放缓、新用户无法激活、留存薄弱、收入停滞，或者实验无法推动业务。症状只能指出需要调查的位置，不能证明原因。

增长诊断协议连接六项决策：

```text
Context -> Outcome -> Constraint -> Evidence -> Action -> Execution
```

目标不是生成最长的想法清单，而是识别当前最重要的约束，说明证据与不确定性，并选择能够改变决策的最小行动。

## 六层诊断结构

| 层级 | 必答问题 | 输出 |
| --- | --- | --- |
| Context | 正在诊断什么产品、客户、商业模式、市场与阶段？ | 有边界的诊断单元 |
| Outcome | 哪项客户或业务结果没有达到预期，影响谁，发生在哪个周期？ | 可衡量目标与基线 |
| Constraint | 当前哪个机制最限制结果？ | 一个主要约束与可能的替代解释 |
| Evidence | 哪些行为、Cohort、定性、经济性与实验数据支持判断？ | 证据账本、冲突与缺失数据 |
| Action | 未来 30 天应该改变什么？ | 优先行动与面向决策的实验 |
| Execution | 工作适合人工完成、使用开源 Skill，还是需要企业级实施？ | 负责人、执行路径、依赖与升级边界 |

## 最小诊断输入

不要从“改善 SEO”或“提高转化”这样的策略请求开始。先记录：

| 字段 | 最少信息 |
| --- | --- |
| 产品 | 产品类型、核心价值与自然使用频率 |
| 客户 | 主要用户、付款人、分群与当前替代方案 |
| 商业模式 | 收入模型、价格或价值指标、主要成本结构 |
| 阶段 | PMF 前、早期可重复、规模化、扩张或成熟优化 |
| 目标结果 | 指标、基线、目标、分群与决策周期 |
| 当前症状 | 观察到的变化、受影响 Cohort 与开始时间 |
| 旅程证据 | 获客、激活、留存、商业化与循环信号 |
| 已采取行动 | 重要改动、实验、Campaign 与已知结果 |
| 约束条件 | 数据、工程、预算、渠道、监管、信任与团队限制 |

如果必填字段未知，应明确记录为缺失证据，不能用虚假精度填补。

## 选择诊断路径

### 获客诊断

适用于合格需求、触达、渠道规模或获客经济性不符合预期的情况。

1. 区分流量规模与合格、留存需求。
2. 按渠道、受众、承诺、入口页面、市场与 Cohort 分群。
3. 对比边际 CAC、回本、增量、激活与留存。
4. 检查渠道模式匹配、饱和、素材或内容供给与归因重叠。
5. 选择无需增加低质量流量即可验证的最大约束。

从主手册[获客](../acquisition/README.zh.md)开始，参考[获客案例](../../case-library/themes/acquisition/README.zh.md)，并使用 [Growth Skills](https://github.com/krillinai/growth-skills) 执行 SEO Audit 等有明确边界的任务。

### 激活诊断

适用于用户已经到达产品，但没有获得首次价值或完成合格下一步行为的情况。

1. 定义用户意图与首次价值事件。
2. 绘制 Setup、Aha 与 Habit，而不是只看注册完成。
3. 找到合格 Cohort 损失最大的转化环节。
4. 把完成率与留存激活、错误、信任、支持和质量一起比较。
5. 测试一个能够减少到达价值时间或不确定性的机制。

从[激活](../activation/README.zh.md)开始，并参考[激活案例](../../case-library/themes/activation/README.zh.md)。

### 留存诊断

适用于用户没有回来、重复价值、召回或加深使用的情况。

1. 选择符合产品自然使用周期的留存间隔。
2. 区分新增、留存、召回、扩展与流失行为。
3. 按意图、激活路径、来源、市场与产品状态比较 Cohort。
4. 判断哪项重复价值、习惯、工作流、网络或服务条件消失。
5. 在增加提醒、折扣或通知数量前修复价值损失。

从[留存](../retention/README.zh.md)开始，并参考[留存案例](../../case-library/themes/retention/README.zh.md)。

### 商业化诊断

适用于使用行为没有转化成健康、可留存收入的情况。

1. 分别定义用户、付款人、受益人和价值指标。
2. 按客户类型拆分转化、收入、留存、扩展与成本。
3. 检查价格、套餐、支付意愿、服务成本与渠道经济性。
4. 区分商业化问题与激活或留存不足。
5. 测试商业改动时保护客户价值与贡献毛利。

从[商业化](../monetization/README.zh.md)开始，并参考[商业化案例](../../case-library/themes/monetization/README.zh.md)。

### 增长循环诊断

适用于推荐、协作、内容、交易或其他产品行为应当产生更多分发或价值的情况。

1. 定义循环输入、参与动机、接收者与下一项价值。
2. 衡量参与率、Payload、转化、周期时间与接收者留存。
3. 区分病毒传播与网络效应，以及总扩散与增量。
4. 纳入奖励成本、滥用、蚕食、信任与平台依赖。
5. 确认循环通过已接收且重复的价值闭合。

从[推荐与增长循环](../referral-and-loops/README.zh.md)开始，并参考[增长循环案例](../../case-library/themes/referral-and-loops/README.zh.md)。

### 增长系统诊断

适用于团队拥有很多想法和活动，却无法持续衡量、决策、执行或学习的情况。

1. 明确客户价值结果与负责人。
2. 绘制决策权、依赖、数据质量、工具与共享能力。
3. 检查实验是否围绕当前约束并最终形成决策。
4. 区分能力缺失、流程薄弱与所有权不清。
5. 只有重复且稳定的工作，并且存在明确内部使用者时，才建设或集中化。

从[增长指标体系](../metrics/README.zh.md)、[实验系统](../experimentation/README.zh.md)、[增长基础设施](../growth-infrastructure/README.zh.md)与[增长组织](../growth-organization/README.zh.md)开始。

## 选择核心约束

约束评分用于规范比较，不用于制造确定性。

| 维度 | 问题 |
| --- | --- |
| 结果影响 | 如果约束移动，目标结果最多能改变多少？ |
| 证据强度 | 判断是否同时得到行为、Cohort、经济性与定性证据支持？ |
| Reach | 影响多少合格用户或系统范围？ |
| 可控制性 | 团队能否在决策周期内改变机制？ |
| 学习价值 | 行动能否解决重要不确定性？ |
| 风险 | 可能产生哪些客户、信任、监管、运营或经济伤害？ |

只选择一个主要约束。记录次要约束，并解释为何不优先处理。

## 证据账本

每次诊断都需要明确证据边界：

| 证据类型 | 示例 | 主要局限 |
| --- | --- | --- |
| 行为 | 事件、漏斗、Cohort、频率、交易 | 埋点与选择错误 |
| 经济性 | CAC、回本、毛利、收入留存、激励成本 | 分摊与预测假设 |
| 定性 | 访谈、支持、销售、观察、开放回答 | 抽样与解释偏差 |
| 实验 | Holdout、随机测试、分阶段发布 | 统计功效、干扰、实现与新鲜感 |
| 外部 | 基准、竞争者、市场与平台数据 | 可比性与时效性 |

应保留矛盾证据。当证据相互冲突或重要分群缺失时，置信度必须下降。

## 诊断输出

人工评审、AI Agent 响应或 clawee.ai 诊断统一使用以下结构：

```text
1. 诊断范围
2. 目标结果与基线
3. 主要约束
4. 支持与矛盾证据
5. 置信度与缺失证据
6. 30 天行动
7. 面向决策的实验
8. 相关 Handbook 概念
9. 相关公司与人物案例
10. 执行路径、负责人、依赖与护栏
```

机器可读协议见 [`protocol.schema.json`](protocol.schema.json)。

## 执行路径

### 自助执行

当团队可以访问必要数据、完成决策，并且无需持续集成即可执行有边界的改动时，直接使用 Playbook。

### Growth Skills

当诊断可以映射成可复用、可审计的工作流时，使用 [krillinai/growth-skills](https://github.com/krillinai/growth-skills)。当前目录包含 [SEO Audit](https://github.com/krillinai/growth-skills/tree/main/skills/seo-audit)，用于形成有证据支持的技术、内容与搜索诊断。

### clawee.ai 企业实施

当实施涉及以下一种或多种条件时，适合使用 [clawee.ai](https://clawee.ai/)：

- 持续访问 CRM、分析、广告、内容、销售或产品系统；
- 跨渠道身份、归因或数据建模；
- 持续 Agent 执行、监控与适应；
- 多团队权限、审批、审计或治理；
- 定制指标、公司私有证据或安全数据处理；
- 多产品、多市场或企业级协调。

企业升级应发生在诊断之后，不能以此隐藏方法或扣留可用的自助结果。

## 复盘节奏

1. 确认目标结果或 Context 是否变化。
2. 复查上次诊断后新增的证据。
3. 判断主要约束是否仍然成立。
4. 关闭、继续或重新设计每项行动与实验。
5. 记录增长模型发生了什么变化，以及下一项决策。

诊断是一项持续维护的决策，不是永久标签。
