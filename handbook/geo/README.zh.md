# 生成式引擎优化（GEO）

> GEO 是一套改善公司、产品或知识在 AI 生成答案中被发现、准确表达与可信引用的实践，并进一步把这种可见性连接到合格客户与业务结果。

**[English](README.md)｜[简体中文](README.zh.md)**

## 定义与边界

| 概念 | 核心目标 | 边界 |
| --- | --- | --- |
| SEO | 通过搜索结果和搜索功能获得合格发现 | 排名与流量不能证明下游客户价值。 |
| GEO | 改善 AI 生成答案中的表达、提及、引用与推荐出现 | AI 可见性不能证明需求、偏好、流量或收入。 |
| Answer Engine Optimization | 让信息适用于直接返回综合答案的系统 | 与 GEO 高度重叠，行业术语尚未标准化。 |
| 内容战略 | 决定哪些客户问题、知识、格式与分发值得投入 | 内容数量本身不能创造权威或检索。 |
| Digital PR 与权威 | 获得可信的独立讨论、引用与声誉 | 缺乏相关性或信任的提及只会增加噪声。 |
| 结构化数据 | 用机器可读形式描述实体与页面内容 | 标记可以减少歧义，但不能保证索引、引用或推荐。 |

GEO 与 SEO 共享基础：可访问页面、清晰实体、有用内容、可信证据、技术质量和外部引用。主要差异在于观察的界面和指标。应把 GEO 视为获客与发现系统的一部分，而不是 SEO 的替代品或一套独立增长战略。

## GEO 在增长系统中的位置

```text
客户问题或决策
-> 搜索、助手或 AI 答案界面
-> 检索与生成的回答
-> 准确提及、引用或推荐
-> 合格下一步行为
-> 激活、留存与商业价值
```

前四步描述可见性。只有当曝光触达相关客户并改变有价值的下游行为时，增长才真正开始。

GEO 可以支持：

- 品类与问题教育；
- 品牌和产品发现；
- 比较与评估；
- 技术或产品支持；
- 声誉与事实准确性；
- 合格访问、产品试用、Pipeline 或合作伙伴兴趣。

它不能弥补薄弱的产品市场匹配、模糊定位、不可信产品或低留存。

## GEO 单位、范围与版本

优化之前先定义测量对象。

| 范围字段 | 决策内容 |
| --- | --- |
| 实体 | 公司、品牌、产品、人物、数据集、品类或主题 |
| 受众 | 客户分群、角色、市场、语言与购买情境 |
| 问题集 | 问题、使用场景、比较、异议、支持与品牌问题 |
| 界面 | AI 助手、AI 搜索、搜索结果答案功能或嵌入式 Copilot |
| 地区与语言 | 市场、Locale、设备、监管与本地术语 |
| 输出类型 | 提及、引用、推荐、答案准确性或合格引荐 |
| 时间窗口 | 收集日期、可观察的模型或产品版本与复盘节奏 |
| 负责人 | 对问题集、证据、变化和业务解读负责的人 |

AI 答案会随平台、模型、检索模式、账户状态、地区、时间与提示词表达变化。任何分数都只是特定收集流程的样本，不是品牌的普遍属性。

## 查询与提示词组合

从真实客户决策建立测量集合，不要只选择容易出现品牌的提示词。

| 意图 | 问题类型示例 | 有用下游信号 |
| --- | --- | --- |
| 问题发现 | 团队如何解决某种重复工作流问题？ | 合格品类兴趣 |
| 品类教育 | 什么是这个品类，什么时候有用？ | 正确理解品类与问题 |
| 方案发现 | 哪些方法或产品可以解决问题？ | 进入相关考虑集合 |
| 比较 | 针对明确场景比较产品 A 与产品 B | 正确差异与评估标准 |
| 证据与信任 | 产品是否可靠、安全、合规或经过验证？ | 使用证据并降低不确定性 |
| 实施 | 应该如何使用产品或方法？ | 激活、支持成功或采用 |
| 品牌准确性 | 公司或产品具体做什么？ | 当前且准确的表达 |

提示词契约应记录：

```text
提示词 ID：
受众与意图：
市场与语言：
提示词文本与受控变体：
平台与界面：
收集方法与日期：
预期实体与事实：
可接受回答边界：
业务相关性：
负责人和复盘日期：
```

使用同义改写组合，而不是单一表达。保留稳定 Benchmark 集用于趋势比较，并使用轮换发现集吸收新的客户语言。分开品牌、非品牌、比较与支持提示词。

## AI 答案界面如何使用信息

AI 产品可能组合多种信息路径：

- 模型训练数据；
- 实时 Web 搜索或检索索引；
- 授权或合作伙伴数据；
- 结构化产品或知识源；
- 用户提供的文件、历史或上下文；
- Tool Call 与第三方集成。

具体组合取决于平台和请求，而且通常无法完整观察。被引用页面可能只支持一项主张，而不是答案的唯一影响来源。没有引用的陈述可能来自训练数据、其他检索来源或模型生成。

除非平台提供了足够证据，或存在可信比较，否则不要声称某个标记字段、爬虫、外链或内容修改直接造成了答案变化。

## 实体与来源基础

系统应该让重要实体和主张容易被识别与验证。

### 实体一致性

- 尽可能统一公司、产品与人物的规范名称。
- 维护清晰的 About、产品、文档、联系方式、政策与支持页面。
- 区分公司、产品家族、单独产品、版本与功能。
- 保持描述、URL、Logo、标识符与所有权关系一致。
- 解释改名、收购、退出或合并的产品。
- 市场存在差异时提供 Locale 专属名称与术语。

### 规范事实

为客户和答案系统反复需要的事实维护唯一可信页面：

- 产品用途与最佳匹配场景；
- 当前能力与限制；
- 公开情况下的价格与套餐；
- 集成、兼容性、安全与合规；
- 作者、发布日期、更新日期与方法；
- 公司历史、管理层、地点与联系方式；
- 政策、支持边界与 Change Log。

相互冲突的官方页面会同时削弱客户信任与机器理解。

## 内容架构

围绕决策与证据建设内容，而不是围绕关键词或提示词数量。

| 内容类型 | 任务 |
| --- | --- |
| 品类与问题指南 | 解释问题、替代方案、决策标准与边界 |
| 产品与使用场景页 | 把具体客户情境连接到能力、价值与证据 |
| 比较页 | 使用明确、公平的标准比较真实替代方案 |
| 文档 | 提供准确的实施、兼容、失败与排错细节 |
| 原创研究 | 提供其他人可以评估和引用的方法、数据、定义与发现 |
| 客户证据 | 说明环境、干预、结果、归因限制与迁移边界 |
| FAQ 或术语表 | 当简短独立答案能改善理解时，解决重复问题 |
| Change Log | 及时更新快速变化的主张、功能与可用性 |

有用的答案型内容通常包含：

1. 直接答案或定义。
2. 主张成立的条件。
3. 支持证据与来源链接。
4. 示例或实施细节。
5. 局限、替代方案与权衡。
6. 清晰作者与新鲜度信号。
7. 对客户有意义的下一步行动。

不要针对合成提示词创建大量近似重复页面。如果客户决策与答案基本相同，应合并内容。

## 证据、引用与主张质量

为重要陈述建立主张账本。

| 字段 | 要求 |
| --- | --- |
| 主张 | 具体陈述内容 |
| 实体与范围 | 产品、市场、分群、版本与时期 |
| 证据 | 第一方数据、文档、研究、客户记录或独立来源 |
| 证据强度 | Direct、supporting、inferred、disputed 或 missing |
| 新鲜度 | 发布日期与最近验证日期 |
| 局限 | 选择、方法、环境、不确定性或迁移边界 |
| 负责人 | 负责修正与复盘的人 |

事实主张优先使用一方文档和原始证据。只有当独立来源增加外部验证、比较或声誉语境时才使用它们。引用真正支持陈述的来源，而不是重复该说法的页面。

避免：

- 缺少证据的最高级与市场领先主张；
- 缺少人群、方法、日期与来源的统计数据；
- 缺少环境或归因限制的客户结果；
- 没有打开验证的生成式引用；
- 把重要局限隐藏在脚注；
- 在不同语言或渠道中改变事实。

## 技术可访问性

技术工作应该让内容可被访问和正确理解，但不能假设存在特殊的 GEO 排名信号。

### 抓取与索引基础

- 规范页面返回稳定的成功响应；
- 使用描述性 URL、Canonical、Title 与标题；
- 重要内容无需不受支持的交互即可访问；
- 维护有效内部链接与 XML Sitemap；
- 避免意外的 `noindex`、认证或渲染阻止；
- 管理重复、参数化、Staging 与过期页面；
- 监控服务器错误、重定向链与内容变化。

### 爬虫策略

爬虫名称与政策会变化。应把搜索或检索访问与模型训练控制分开，一项指令不一定同时控制两者。查阅各平台当前官方文档，并记录每项规则背后的业务、版权、隐私、带宽与安全决策。

`robots.txt` 是声明式访问政策，不是安全控制。敏感数据绝不能依赖爬虫遵守规则来保护。

### 结构化数据

使用与页面可见内容相符的 Schema.org 类型，例如适用时使用 `Organization`、`Product`、`SoftwareApplication`、`Article`、`Person`、`FAQPage` 或 `HowTo`。验证语法，并确保标记与页面一致。

结构化数据可以改善实体清晰度和受支持搜索功能的资格，但不能保证 AI 引用或推荐。

### 新兴文件与协议

`llms.txt` 等格式可以作为实验性文档或发现辅助，但并非普遍标准，也没有被证明是排名控制。只有在假设明确、维护成本较低时采用，不能期待它替代可抓取页面、文档或 Sitemap。

## 站外权威与分发

AI 答案可能使用官方域名之外的来源。应在相关社区与评估者已经工作的地方提供有用证据，从而建立权威。

可能的来源包括：

- 可信行业出版物与研究；
- 合作伙伴和集成文档；
- 软件目录与应用市场；
- 专家评测与比较资源；
- 会议演讲、Podcast、视频与 Transcript；
- 公共代码仓库、标准工作与技术社区；
- 客户讨论与支持社区；
- 原始数据集与透明研究方法。

目标不是最大化提及数量。优先相关性、独立性、事实一致性、可发现性与受众信任。不要使用虚假评论、未披露赞助、自动论坛 Spam 或引用操纵。

## 定位与信息一致性

GEO 会暴露定位质量。如果官方页面、客户、评测者和产品文档对产品的描述不同，答案系统可能重复这种冲突。

连接：

```text
最佳客户与问题
-> 竞争替代方案
-> 差异化价值
-> 能力与证据
-> 一致的产品、内容与外部证明
```

公司尚未说明最适合哪种客户与情境时，不应先优化宽泛的“最佳产品”提示词。完整定位系统参见[定位](../positioning/README.zh.md)。

## 国际化与多语言 GEO

翻译只是市场覆盖的一部分。需要验证：

- 本地客户问题与术语；
- 品类含义与竞争替代方案；
- 本地搜索与 AI 界面；
- 产品可用性、价格、支付与支持；
- 监管、隐私与文化约束；
- 本地证据、引用、社区与可信来源；
- 不同语言版本的实体一致性；
- hreflang、Canonical、Locale 与语言识别行为。

关键决策内容应由母语使用者复核。建立全球汇总分数之前，先分别衡量每个市场。

## 衡量契约

| 指标 | 定义 | 主要局限 |
| --- | --- | --- |
| 提示词覆盖率 | 成功收集有效答案的合格追踪提示词比例 | 依赖提示词样本与收集成功率 |
| 提及率 | 提及目标实体的答案比例 | 提及可能无关、负面或错误 |
| 引用率 | 引用自有或相关外部来源的答案比例 | 各平台展示引用的方式不同 |
| 引用份额 | 实体引用量除以定义竞争集合的引用量 | 竞争对手与提示词定义会改变结果 |
| 推荐出现率 | 相关方案提示词中把实体列为选项的比例 | 被列入不代表偏好或转化 |
| 答案准确率 | 被审计事实主张中正确且当前有效的比例 | 需要持续维护事实集合与人工复核 |
| 定位一致度 | 使用预期品类、客户、价值与证据的答案比例 | 预期表达本身也可能错误 |
| 情感或立场 | 对实体的语气与陈述编码 | 自动情感分析可能忽略环境与混合评价 |
| 波动率 | 使用同一契约重复收集时的变化 | 模型与检索变化可能无法由公司控制 |
| 合格引荐 | 可归因于可识别 AI 界面的相关访问或行为 | Referrer 与归因数据不完整 |
| 辅助结果 | 观察到 AI Touch 的 Pipeline、注册、激活或收入 | 没有比较时不能证明增量 |

报告分子、分母、提示词集、平台、市场、语言、收集日期、重试、缺失答案与编码方法。始终把可见性指标与客户和业务结果分开。

## 收集与评估设计

1. 固定 Benchmark 提示词集与受控变体。
2. 定义平台、模式、市场、语言、账户和收集时间。
3. 在条款允许时保存原始回答、引用、时间戳与收集元数据。
4. 规范实体别名，但不能隐藏有歧义的匹配。
5. 使用书面规则编码提及、引用、推荐、准确性、定位与情感。
6. 对一部分样本进行双人复核，并衡量主观标签分歧。
7. 保留失败、拒绝、缺失引用与互相矛盾的答案。
8. 在隐私和归因允许时连接可识别引荐与下游结果。
9. 当平台、提示词、产品或编码规则变化时更新契约版本。

不要反复运行提示词直到获得满意答案，然后只报告有利样本。

## 实验与因果设计

AI 平台很少提供由公司完全控制的干净实验界面，应使用当前可行的最强设计：

| 问题 | 可用设计 |
| --- | --- |
| 页面是否更容易被检索或引用？ | 在提示词与抓取控制稳定时分批更新页面 |
| 改善证据是否改变答案准确性？ | 保持事实、提示词和编码规则不变的前后审计 |
| 来源分发项目是否增加合格可见性？ | 匹配主题、市场或时期，并设置下游护栏 |
| AI 可见性是否创造增量需求？ | 在曝光能够有效分离时使用地域、受众或时间 Holdout |
| 哪种内容模式表现更好？ | 使用预先声明资格与观察窗口的可比内容族 |

平台更新、模型变化、抓取延迟、提示词漂移、竞争活动与季节性都是重要混杂因素。观察变化只能用于指导调查，不能直接证明因果。

## 诊断地图

| 观察到的模式 | 下一步检查 | 可能响应 |
| --- | --- | --- |
| 相关提示词没有提及 | 实体清晰度、抓取、内容覆盖、外部证据 | 增加提示词数量之前先修复基础。 |
| 已有提及但事实错误 | 规范事实、过期页面、冲突来源、实体混淆 | 修正唯一可信页面，并记录矛盾来源。 |
| 自有页面可见但没有引用 | 主张质量、独立支持、回答格式、平台引用行为 | 改善证据，同时接受引用展示由平台决定。 |
| 品牌提示词表现好，非品牌弱 | 品类与问题内容、定位、外部权威 | 建设与需求相关的知识，而不是更多品牌页。 |
| 可见性上升但合格结果不变 | 提示词相关性、受众、推荐环境、下一步行动 | 重建提示词组合或改善回答后的路径。 |
| 某种语言表现薄弱 | 本地术语、市场可用性、本地来源、技术 Locale 信号 | 本地化决策语境与证据，而不只是文字。 |
| 多次运行结果波动很大 | 采样、检索模式、模型变化、提示词歧义、证据强度 | 增加重复样本并报告波动。 |
| 竞争对手主导比较提示词 | 真实替代方案、决策标准、证据、第三方来源 | 改善定位与可信比较证据。 |
| 大量引用来自薄弱页面 | 来源质量、重复、转载、过期 URL | 合并规范来源，提高站外分发质量。 |
| 已允许爬虫但可见性仍弱 | 内容价值、权威、查询匹配、平台行为 | 停止把爬虫访问当作排名机制。 |

## 运营节奏

### 每月

- 收集稳定 Benchmark 并复盘波动；
- 审计重要事实错误与新增引用；
- 检查技术访问、索引与来源新鲜度；
- 把可见性变化连接到合格引荐与下游证据；
- 分配修正、内容或分发工作。

### 每季度

- 更新轮换提示词发现集；
- 复盘定位、品类、竞争、市场与语言覆盖；
- 退出低价值提示词和重复内容；
- 评估来源多样性与证据质量；
- 根据官方文档更新爬虫与平台政策；
- 决定哪些 GEO 工作应该扩大、改变或停止。

## 成熟度模型

| 阶段 | 特征 | 下一项约束 |
| --- | --- | --- |
| 1. 零散检查 | 员工人工询问少量容易出现品牌的提示词 | 定义实体、客户问题与可复现收集契约。 |
| 2. 可见性基线 | 追踪稳定提示词、平台、提及与引用 | 加入答案准确性、来源质量、市场和业务相关性。 |
| 3. 证据支持的项目 | 内容、技术、实体与权威工作遵循明确假设 | 把可见性连接到合格行为和更强比较。 |
| 4. 受治理的运营系统 | 共享负责人、版本、主张账本、复盘与跨职能决策 | 管理平台变化、国际覆盖与因果不确定性。 |
| 5. 自适应发现系统 | GEO、SEO、内容、PR、产品证据与客户结果共同更新 | 在界面和产品变化中保持信任与可比性。 |

成熟度不取决于提示词、页面、工具或提及数量，而取决于团队能否在保留证据、信任与下游价值的情况下改善准确客户发现。

## 建设顺序

1. 定义实体、客户、市场、语言、决策与业务结果。
2. 从客户、搜索、销售、支持与产品证据建立稳定提示词组合。
3. 建立规范事实、实体关系、负责人和更新日期。
4. 审计抓取、索引、渲染、重复、结构化数据与爬虫策略。
5. 梳理品类、场景、比较、证明、文档与支持内容缺口。
6. 建立主张账本，并改善每项重要事实的最强来源。
7. 在不操纵的前提下获得相关独立证据与分发。
8. 收集可复现的可见性与准确性基线。
9. 把可识别 AI 引荐与辅助结果连接到激活、留存与经济性。
10. 运行有边界测试，记录混杂因素，并按月和季度复盘。

## 护栏

- 不伪造引用、评论、客户、专家或研究。
- 不发布虚假或误导内容操纵答案系统。
- 不把模型训练控制与搜索访问视为同一决策。
- 不为可见性暴露隐私、版权、监管或安全敏感信息。
- 不创建缺少独立客户价值的 Doorway Page 或近似重复提示词页面。
- 不在缺少提示词集与方法时报告专有可见性分数。
- 不从一个答案、一个模型或一次前后观察推断因果。
- 不优化会吸引产品无法服务客户的提及。

## 精选研究与官方参考

- [GEO: Generative Engine Optimization](https://arxiv.org/abs/2311.09735)：提出 Benchmark，并在特定实验环境中评估内容干预；结果不能直接推广到所有平台或查询。
- [Adversarial Search Engine Optimization for Large Language Models](https://arxiv.org/abs/2406.18382)：研究 LLM 搜索中的操纵风险，支持建立信任与滥用护栏。
- [Manipulating Large Language Models to Increase Product Visibility](https://arxiv.org/abs/2404.07981)：研究策略性文本如何改变产品推荐，同时揭示衡量机会与诚信风险。
- [Google 搜索与 AI 功能](https://developers.google.com/search/docs/appearance/ai-features)：Google 面向网站所有者的当前 AI 搜索功能官方指南。
- [OpenAI 爬虫文档](https://developers.openai.com/api/docs/bots)：当前爬虫身份与控制指南。
- [Perplexity 爬虫文档](https://docs.perplexity.ai/docs/resources/perplexity-crawlers)：当前爬虫身份与访问指南。

研究结果、爬虫身份、产品行为与平台政策都会变化。进行运营调整前应重新核对当前官方文档。

## 证据与局限

- 答案系统部分不透明且频繁变化，观察到的可见性不能揭示所有检索或生成机制。
- 提示词追踪只是对决策空间的采样，无法代表所有用户或个性化环境。
- 引用、提及、推荐与情感是平台专属概念，没有统一指标。
- 可见性可能因模型、索引、竞争、市场或平台变化而改变，与公司行动无关。
- AI 引荐可能被隐藏、跨设备、延迟或没有点击，因此业务归因不完整。
- GEO 方法应该被视为有版本的假设，并明确记录证据与不确定性。
