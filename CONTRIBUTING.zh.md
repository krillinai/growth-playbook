# 参与 Growth Playbook

**[English](CONTRIBUTING.md)｜[简体中文](CONTRIBUTING.zh.md)**

Growth Playbook 接受对增长基础、生命周期环节、增长系统、公司案例、人物档案、诊断与工具的有证据支持的改进。开始写作前，请阅读[编辑和证据标准](EDITORIAL.zh.md)。

## 开始之前

出现以下情况时，先提交内容提案 Issue：

- 新增人物、公司、主题或顶层知识模块；
- 实质修改现有模块的分类或边界；
- 引入依赖争议、私有或难以核实证据的主张；
- 增加自动化、生成内容或新的外部依赖。

小范围纠错、表达改进、来源升级、链接修复和边界明确的补充，可以直接提交 Pull Request。

## 内容流程

1. 明确这项贡献需要改善的决策。
2. 找到负责该内容的现有知识、主题、公司或人物页面。
3. 优先使用一手来源，并保留发布日期、角色、公司、产品和市场背景。
4. 区分已核实事实、参与者讲述、公司主张、编辑综合与因果结论。
5. 在被约束的结论附近写出局限与迁移边界。
6. 用相同信息结构更新英文与中文页面。
7. 更新所有相关索引和双向链接。
8. 提交 Pull Request 前运行 `ruby scripts/validate-content.rb`。

## 仓库边界

- `handbook/` 负责定义、模型、指标、诊断、运营方法、成熟度与局限。
- `case-library/themes/` 负责五家公司证据对照与迁移边界。
- `case-library/companies/` 负责同一公司跨多个增长主题的证据。
- `case-library/people/` 负责人物方法、来源索引、局限及人物专属 Skill。
- `toolkit/` 负责工具目录和面向决策的使用指导。
- `docs/` 不属于公开仓库，贡献中不能包含该目录文件。

不要创建空栏目、占位页面、不支持的语言版本，或在内容完整前增加新的顶层分类。

## 双语要求

英文页面使用 `README.md`，简体中文页面使用 `README.zh.md`。内容变化必须保持：

- 相同的章节结构与结论；
- 一致的来源清单与证据边界；
- 成对页面之间语言正确的链接；
- 产品名、公司名、公式与技术术语准确，而不是机械翻译。

翻译可以调整句式，但不能增加或删除实质主张。

## Pull Request 检查清单

- 贡献符合仓库范围。
- 主张遵循 `EDITORIAL.zh.md` 中的证据标签与来源优先级。
- 定义保留在知识模块，公司证据保留在案例资料库。
- 每个主题恰好包含五个详细公司案例。
- 中英文结构一致。
- 索引总数和 `查看全部` 数字准确。
- 本地路径、锚点与 Markdown 表格通过校验。
- 不包含生成产物、私有数据、密钥或 `docs/` 文件。

## 开源许可

提交贡献即表示你同意该内容使用仓库的 [MIT License](LICENSE)。只提交你有权分享的材料，并对受版权保护内容保持少量引用和清晰署名。
